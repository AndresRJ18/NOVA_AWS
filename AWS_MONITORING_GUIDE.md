## Prerequisites

Before you begin, ensure you have:

- ✅ **AWS Account** with access to Amazon Bedrock
- ✅ **IAM Permissions** to view CloudWatch metrics and billing information
- ✅ **Nova Sonic Integration** deployed and running
- ✅ **AWS Console Access** with appropriate credentials

### Required IAM Permissions

To view metrics and billing, your IAM user/role needs these permissions:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "bedrock:GetUsageMetrics",
        "cloudwatch:GetMetricStatistics",
        "cloudwatch:ListMetrics",
        "ce:GetCostAndUsage",
        "budgets:ViewBudget",
        "budgets:ModifyBudget"
      ],
      "Resource": "*"
    }
  ]
}
```

## Accessing Bedrock Usage Metrics


### Step-by-Step Instructions

#### 1. Log in to AWS Console

1. Navigate to [AWS Console](https://console.aws.amazon.com)
2. Sign in with your AWS account credentials
3. Ensure you're in the correct AWS region (top-right corner)
   - Common regions: `us-east-1`, `us-west-2`, `eu-west-1`
   - **Important**: Select the same region where your Nova Sonic integration is deployed

#### 2. Navigate to Amazon Bedrock

**Option A: Using the Search Bar (Fastest)**
1. Click the search bar at the top of the AWS Console
2. Type "Bedrock"
3. Click on "Amazon Bedrock" in the search results

**Option B: Using the Services Menu**
1. Click "Services" in the top navigation bar
2. Under "Machine Learning", find and click "Amazon Bedrock"
3. Or navigate to: [Amazon Bedrock Console](https://console.aws.amazon.com/bedrock)

#### 3. Access Usage Metrics

1. In the Amazon Bedrock console, look at the left sidebar
2. Click on **"Monitoring"** (under the "Management" section)
3. Select **"Usage metrics"** from the submenu
4. You'll see the usage dashboard with various metrics

**Direct Link**: `https://console.aws.amazon.com/bedrock/home?region=YOUR_REGION#/monitoring/usage`
- Replace `YOUR_REGION` with your AWS region (e.g., `us-east-1`)

### What You'll See

The Usage Metrics dashboard displays:

- **Invocations**: Total number of API calls to Nova Sonic
- **Input Tokens**: Amount of audio input processed (measured in tokens)
- **Output Tokens**: Amount of audio/text generated (measured in tokens)
- **Errors**: Failed API calls and error types
- **Throttles**: Requests that were rate-limited

## Filtering by Model


### Filtering Metrics by Nova Sonic Model

To view metrics specifically for Nova Sonic (and not other Bedrock models):

#### Step 1: Access the Filters

1. On the Usage Metrics page, look for the **"Filters"** section at the top
2. You'll see filter options for:
   - Time range
   - Model ID
   - Operation type

#### Step 2: Filter by Model ID

1. Click on the **"Model ID"** dropdown
2. Type or select: `amazon.nova-sonic-v1:0`
   - If you're using Nova Sonic v2, select: `amazon.nova-2-sonic-v1:0`
3. Click **"Apply"** or press Enter

#### Step 3: Adjust Time Range

1. Click on the **"Time range"** dropdown
2. Select your desired time period:
   - Last 1 hour
   - Last 24 hours
   - Last 7 days
   - Last 30 days
   - Custom range (specify start and end dates)
3. Click **"Apply"**

### Viewing Detailed Metrics

After filtering, you can view:

#### Invocation Metrics
- **Total Invocations**: Number of times Nova Sonic was called
- **Successful Invocations**: Requests that completed successfully
- **Failed Invocations**: Requests that returned errors
- **Invocation Rate**: Requests per minute/hour

#### Token Usage Metrics
- **Input Tokens**: Audio input processed by Nova Sonic
  - For speech-to-text: Audio duration converted to tokens
  - Approximately 1 second of audio ≈ 50 tokens
- **Output Tokens**: Audio/text generated by Nova Sonic
  - For text-to-speech: Audio duration generated
  - For speech-to-text: Text length in tokens

#### Performance Metrics
- **Latency**: Average response time for API calls
  - P50 (median), P90, P95, P99 percentiles
  - Target: <2s for STT, <3s for TTS
- **Error Rate**: Percentage of failed requests
  - Target: <1% error rate

### Exporting Metrics

To export metrics for analysis:

1. Click the **"Actions"** button (top-right)
2. Select **"Download as CSV"**
3. Choose your date range and metrics
4. Click **"Download"**

## Understanding Usage Metrics


### Key Metrics Explained

#### 1. Invocations

**What it measures**: Total number of API calls to Nova Sonic

**Why it matters**: 
- Indicates usage volume and user engagement
- Helps identify usage patterns (peak hours, days)
- Useful for capacity planning

**Typical values**:
- Low usage: <100 invocations/day
- Medium usage: 100-1,000 invocations/day
- High usage: >1,000 invocations/day

#### 2. Input Tokens

**What it measures**: Amount of audio input processed (for speech-to-text)

**Why it matters**:
- Directly impacts costs (charged per input token)
- Indicates how much users are speaking
- Helps optimize recording duration limits

**Conversion**:
- ~50 tokens per second of audio
- 1 minute of audio ≈ 3,000 tokens
- 5 minutes of audio ≈ 15,000 tokens

#### 3. Output Tokens

**What it measures**: Amount of audio/text generated (for text-to-speech)

**Why it matters**:
- Directly impacts costs (charged per output token, higher rate than input)
- Indicates how much the system is speaking
- Helps optimize response length

**Conversion**:
- ~50 tokens per second of audio
- 1 minute of audio ≈ 3,000 tokens
- Text-to-speech: ~4 tokens per word

#### 4. Errors

**What it measures**: Failed API calls and error types

**Why it matters**:
- Indicates system health and reliability
- Helps identify configuration issues
- Triggers for fallback to text mode

**Common error types**:
- `ThrottlingException`: Rate limit exceeded
- `ValidationException`: Invalid input format
- `ServiceUnavailableException`: Bedrock service issue
- `AccessDeniedException`: IAM permission issue

#### 5. Throttles

**What it measures**: Requests that were rate-limited

**Why it matters**:
- Indicates if you're hitting API limits
- May require requesting quota increase
- Affects user experience (delays, failures)

**Default limits**:
- Varies by region and account type
- Check: [Bedrock Service Quotas](https://docs.aws.amazon.com/bedrock/latest/userguide/quotas.html)

## Viewing Costs in AWS Billing Dashboard


### Step-by-Step Instructions

#### 1. Navigate to AWS Billing Dashboard

**Option A: Direct Link**
- Go to: [AWS Billing Dashboard](https://console.aws.amazon.com/billing/home)

**Option B: Via Console**
1. Click your account name in the top-right corner
2. Select **"Billing and Cost Management"** from the dropdown
3. You'll be redirected to the Billing Dashboard

#### 2. View Current Month Costs

1. On the Billing Dashboard home page, you'll see:
   - **Month-to-Date Costs**: Current month's spending
   - **Forecasted Costs**: Estimated end-of-month total
   - **Last Month's Costs**: Previous month's final bill

2. Look for the **"Cost by Service"** section
3. Find **"Amazon Bedrock"** in the list
4. Click on it to see detailed breakdown

#### 3. View Detailed Cost Breakdown

1. In the left sidebar, click **"Cost Explorer"**
2. If prompted, enable Cost Explorer (free, but takes 24 hours to activate)
3. Once enabled, you'll see a graph of your costs over time

#### 4. Filter by Amazon Bedrock

1. In Cost Explorer, click **"Filters"** on the right side
2. Under **"Service"**, select **"Amazon Bedrock"**
3. Click **"Apply filters"**

#### 5. View Costs by Time Period

**Daily Costs**:
1. Set **"Granularity"** to **"Daily"**
2. Select date range (e.g., last 7 days, last 30 days)
3. View daily spending trend

**Weekly Costs**:
1. Set **"Granularity"** to **"Weekly"**
2. Select date range (e.g., last 4 weeks, last 12 weeks)
3. View weekly spending trend

**Monthly Costs**:
1. Set **"Granularity"** to **"Monthly"**
2. Select date range (e.g., last 3 months, last 6 months)
3. View monthly spending trend

#### 6. View Cost by Model

To see costs specifically for Nova Sonic:

1. In Cost Explorer, click **"Group by"**
2. Select **"Usage Type"**
3. Look for entries containing:
   - `Bedrock-InvokeModel-Input-NovaS onic` (input tokens)
   - `Bedrock-InvokeModel-Output-NovaSonic` (output tokens)

### Understanding Your Bill

#### Cost Components

**Amazon Bedrock Nova Sonic Pricing** (as of 2024):

| Component | Price | Notes |
|-----------|-------|-------|
| Input Tokens | ~$0.003 per 1,000 tokens | Audio input (speech-to-text) |
| Output Tokens | ~$0.012 per 1,000 tokens | Audio output (text-to-speech) |

**Example Calculation**:

```
Scenario: 100 interview sessions per month
- Average session: 10 voice interactions
- Average input: 30 seconds per interaction (1,500 tokens)
- Average output: 20 seconds per interaction (1,000 tokens)

Monthly Usage:
- Input tokens: 100 sessions × 10 interactions × 1,500 tokens = 1,500,000 tokens
- Output tokens: 100 sessions × 10 interactions × 1,000 tokens = 1,000,000 tokens

Monthly Cost:
- Input cost: 1,500,000 ÷ 1,000 × $0.003 = $4.50
- Output cost: 1,000,000 ÷ 1,000 × $0.012 = $12.00
- Total: $16.50/month
```

#### Cost Breakdown by Operation

**Speech-to-Text (STT)**:
- Charges: Input tokens only
- Cost: ~$0.003 per 1,000 tokens
- Example: 1 minute of audio = 3,000 tokens = $0.009

**Text-to-Speech (TTS)**:
- Charges: Output tokens only
- Cost: ~$0.012 per 1,000 tokens
- Example: 1 minute of audio = 3,000 tokens = $0.036

**Note**: Output tokens are 4x more expensive than input tokens!

### Downloading Cost Reports

1. In the Billing Dashboard, click **"Bills"** in the left sidebar
2. Select the month you want to review
3. Click **"Download CSV"** to get detailed line items
4. Open in Excel/Google Sheets for analysis

## Setting Up Budget Alerts


### Why Set Up Budget Alerts?

Budget alerts help you:
- ✅ Prevent unexpected charges
- ✅ Monitor spending in real-time
- ✅ Get notified before costs exceed limits
- ✅ Take action before bills get too high

### Step-by-Step Instructions

#### 1. Navigate to AWS Budgets

**Option A: Direct Link**
- Go to: [AWS Budgets Console](https://console.aws.amazon.com/billing/home#/budgets)

**Option B: Via Billing Dashboard**
1. Go to [AWS Billing Dashboard](https://console.aws.amazon.com/billing/home)
2. Click **"Budgets"** in the left sidebar
3. You'll see your existing budgets (if any)

#### 2. Create a New Budget

1. Click the **"Create budget"** button (orange button, top-right)
2. You'll see budget type options

#### 3. Select Budget Type

Choose **"Cost budget"** (recommended for Nova Sonic monitoring):
- Tracks total spending across services
- Most flexible for general cost monitoring

Other options:
- **Usage budget**: Tracks specific usage metrics (e.g., tokens)
- **Savings Plans budget**: For reserved capacity
- **Reservation budget**: For reserved instances

Click **"Next"** after selecting "Cost budget"

#### 4. Configure Budget Details

**Budget Name**:
- Enter a descriptive name: `Nova-Sonic-Monthly-Budget`
- Or: `Bedrock-Cost-Alert-$100`

**Period**:
- Select **"Monthly"** (recommended)
- Other options: Daily, Quarterly, Annually

**Budget Amount**:
- **Fixed**: Set a specific amount (e.g., $100/month)
- **Planned**: Different amounts for different months
- Choose **"Fixed"** and enter your threshold (e.g., `100`)

**Budget Scope**:
- Select **"Filtered"** to track only Bedrock costs
- Under **"Service"**, select **"Amazon Bedrock"**
- This ensures alerts are specific to Nova Sonic usage

Click **"Next"**

#### 5. Configure Alert Thresholds

Set up multiple alerts at different thresholds:

**Alert 1: 50% of Budget**
1. Click **"Add an alert threshold"**
2. Set **"Threshold"** to `50` %
3. Select **"Actual"** (not forecasted)
4. Enter email address(es) to notify
5. Click **"Add"**

**Alert 2: 80% of Budget**
1. Click **"Add an alert threshold"**
2. Set **"Threshold"** to `80` %
3. Select **"Actual"**
4. Enter email address(es)
5. Click **"Add"**

**Alert 3: 100% of Budget**
1. Click **"Add an alert threshold"**
2. Set **"Threshold"** to `100` %
3. Select **"Actual"**
4. Enter email address(es)
5. Click **"Add"**

**Alert 4: Forecasted 100% (Optional)**
1. Click **"Add an alert threshold"**
2. Set **"Threshold"** to `100` %
3. Select **"Forecasted"** (predicts if you'll exceed budget)
4. Enter email address(es)
5. Click **"Add"**

**Recommended Alert Configuration**:
```
✅ 50% Actual - Early warning
✅ 80% Actual - Take action soon
✅ 100% Actual - Budget exceeded
✅ 100% Forecasted - Predicted to exceed
```

#### 6. Configure Notification Settings

**Email Notifications**:
1. Enter email addresses (comma-separated for multiple)
2. Example: `admin@company.com, devops@company.com`
3. AWS will send a confirmation email to each address
4. Recipients must click the confirmation link to receive alerts

**SNS Topic (Advanced)**:
- Optionally, create an SNS topic for programmatic alerts
- Useful for integrating with Slack, PagerDuty, etc.
- Click **"Create a new topic"** if needed

#### 7. Review and Create

1. Review all budget settings:
   - Budget name
   - Amount
   - Period
   - Filters (Amazon Bedrock)
   - Alert thresholds
   - Email addresses
2. Click **"Create budget"**
3. You'll see a confirmation message

#### 8. Confirm Email Subscriptions

1. Check your email inbox
2. Look for emails from `no-reply@sns.amazonaws.com`
3. Click **"Confirm subscription"** in each email
4. You'll see a confirmation page

**Important**: Alerts won't be sent until you confirm the subscription!

## Cost Optimization Strategies


### 1. Enable Audio Caching

**Impact**: Reduces API calls by 60%+ for common phrases

**How to implement**:
- Set `AUDIO_CACHE_SIZE_MB=100` (or higher) in environment variables
- Cache common phrases: greetings, instructions, transitions
- Cache TTL: 24 hours for static content, 1 hour for dynamic

**Example savings**:
```
Without caching: 1,000 TTS requests/day × $0.036 = $36/day
With 60% cache hit: 400 TTS requests/day × $0.036 = $14.40/day
Savings: $21.60/day = $648/month
```

### 2. Optimize Recording Duration

**Impact**: Reduces input token costs

**How to implement**:
- Set `MAX_AUDIO_DURATION_SECONDS=120` (2 minutes)
- Encourage concise responses
- Implement auto-stop after silence detection

**Example savings**:
```
Average 60s recording: 3,000 tokens × $0.003 = $0.009
Optimized 30s recording: 1,500 tokens × $0.003 = $0.0045
Savings: 50% reduction in input costs
```

### 3. Use Development Mode for Testing

**Impact**: Zero API costs during development

**How to implement**:
- Set `ENABLE_DEV_MODE=true` for development environment
- Use mock audio files instead of Nova Sonic API
- Only use production API for staging/production

**Example savings**:
```
Development testing: 500 requests/day × $0.05 = $25/day
With dev mode: $0/day
Savings: $25/day = $750/month
```

### 4. Implement Rate Limiting

**Impact**: Prevents abuse and runaway costs

**How to implement**:
- Set `RATE_LIMIT_MAX_REQUESTS=100` per minute
- Monitor for unusual usage patterns
- Block suspicious IPs

**Example protection**:
```
Without rate limiting: Potential unlimited usage
With rate limiting: Max 100 req/min = 144,000 req/day
Cost cap: 144,000 × $0.05 = $7,200/day (worst case)
```

### 5. Monitor and Optimize Latency

**Impact**: Reduces timeout retries and wasted API calls

**How to implement**:
- Target: <2s for STT, <3s for TTS
- Use AWS region closest to users
- Optimize audio format conversion
- Monitor error rates and retry logic

**Example savings**:
```
High error rate (10%): 1,000 requests + 100 retries = 1,100 API calls
Optimized (1% errors): 1,000 requests + 10 retries = 1,010 API calls
Savings: 90 fewer API calls = ~$4.50/day
```

### 6. Choose the Right Model

**Impact**: Balance cost vs. quality

**Models**:
- `amazon.nova-sonic-v1:0`: Standard quality, lower cost (default)
- `amazon.nova-2-sonic-v1:0`: Higher quality, higher cost (~2x)

**When to use each**:
- **v1**: General interviews, practice sessions, high volume
- **v2**: Premium features, critical interviews, low volume

### 7. Implement Text Fallback

**Impact**: Reduces costs when voice isn't necessary

**How to implement**:
- Allow users to switch to text mode
- Auto-fallback on errors (prevents retry costs)
- Save user preference (text users = $0 voice costs)

**Example savings**:
```
100% voice users: 1,000 sessions × $0.50 = $500
70% voice, 30% text: 700 sessions × $0.50 = $350
Savings: $150/month
```

### 8. Compress Audio Efficiently

**Impact**: Reduces bandwidth and processing time

**How to implement**:
- Use Opus codec (48kbps) instead of MP3 (64kbps)
- Maintain voice quality while reducing size
- Faster transmission = lower latency

**Example savings**:
```
MP3 (64kbps): 1 minute = 480KB
Opus (48kbps): 1 minute = 360KB
Bandwidth savings: 25% reduction
```

### 9. Set Up Automated Monitoring

**Impact**: Early detection of cost spikes

**How to implement**:
- Use AWS CloudWatch alarms
- Monitor daily spending trends
- Alert on anomalies (>2x normal usage)
- Integrate with Slack/email for notifications

### 10. Regular Cost Reviews

**Impact**: Identify optimization opportunities

**How to implement**:
- Weekly: Review usage metrics
- Monthly: Analyze cost trends
- Quarterly: Evaluate model choice and caching strategy
- Annually: Negotiate AWS Enterprise Discount Program (if high volume)

## Troubleshooting


### Common Issues and Solutions

#### Issue 1: No Metrics Showing in Bedrock Console

**Symptoms**:
- Usage metrics page is empty
- No data in graphs

**Possible Causes**:
1. Wrong AWS region selected
2. No API calls made yet
3. Metrics delay (up to 15 minutes)

**Solutions**:
1. Verify you're in the correct region (top-right corner)
2. Make a test API call to Nova Sonic
3. Wait 15-30 minutes for metrics to appear
4. Check CloudWatch Logs for API call evidence

#### Issue 2: Can't Access Billing Dashboard

**Symptoms**:
- "Access Denied" error when viewing billing
- Can't see cost information

**Possible Causes**:
1. IAM user lacks billing permissions
2. Billing access not enabled for IAM users

**Solutions**:
1. Contact AWS account owner to grant permissions
2. Account owner must enable IAM access to billing:
   - Go to Account Settings
   - Enable "IAM User and Role Access to Billing Information"
3. Add billing permissions to IAM policy (see Prerequisites)

#### Issue 3: Budget Alerts Not Received

**Symptoms**:
- Budget exceeded but no email received
- Alerts configured but not working

**Possible Causes**:
1. Email subscription not confirmed
2. Emails going to spam folder
3. Wrong email address configured

**Solutions**:
1. Check email for confirmation link from AWS SNS
2. Click "Confirm subscription" in the email
3. Check spam/junk folder for AWS emails
4. Add `no-reply@sns.amazonaws.com` to safe senders
5. Verify email address in budget settings

#### Issue 4: Unexpected High Costs

**Symptoms**:
- Costs higher than expected
- Sudden spike in usage

**Possible Causes**:
1. High error rate causing retries
2. No caching enabled
3. Abuse or bot traffic
4. Long recording durations

**Solutions**:
1. Check error metrics in Bedrock console
2. Enable audio caching (`AUDIO_CACHE_SIZE_MB=100`)
3. Implement rate limiting (`RATE_LIMIT_MAX_REQUESTS=100`)
4. Set max recording duration (`MAX_AUDIO_DURATION_SECONDS=120`)
5. Review CloudWatch Logs for unusual patterns
6. Block suspicious IPs

#### Issue 5: Metrics Don't Match Expected Usage

**Symptoms**:
- Token count seems wrong
- Invocation count doesn't match logs

**Possible Causes**:
1. Metrics delay (up to 15 minutes)
2. Multiple regions in use
3. Retries counted as separate invocations
4. Caching reducing API calls

**Solutions**:
1. Wait for metrics to update (refresh after 15-30 minutes)
2. Check all regions where Bedrock is deployed
3. Review retry logic in application logs
4. Compare CloudWatch Logs with Bedrock metrics
5. Account for cache hits (not counted in metrics)

#### Issue 6: Can't Filter by Nova Sonic Model

**Symptoms**:
- Model filter doesn't show Nova Sonic
- No data when filtering by model

**Possible Causes**:
1. Model not available in selected region
2. No API calls made to that model yet
3. Using wrong model ID

**Solutions**:
1. Verify Nova Sonic is available in your region:
   - [Bedrock Model Availability](https://docs.aws.amazon.com/bedrock/latest/userguide/models-regions.html)
2. Check environment variable `NOVA_SONIC_MODEL_ID`
3. Make a test API call to the model
4. Wait 15 minutes for metrics to populate
5. Try filtering by "All models" first to see if any data appears

#### Issue 7: Cost Explorer Not Available

**Symptoms**:
- "Cost Explorer is not enabled" message
- Can't access cost analysis features

**Possible Causes**:
1. Cost Explorer not activated
2. Takes 24 hours to activate after enabling

**Solutions**:
1. Enable Cost Explorer:
   - Go to Billing Dashboard
   - Click "Cost Explorer"
   - Click "Enable Cost Explorer"
2. Wait 24 hours for data to populate
3. Use "Bills" page for immediate cost information
